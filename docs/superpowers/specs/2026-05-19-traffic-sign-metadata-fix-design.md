# Traffic-Sign Metadata Fix — Official-Source Pipeline

**Date:** 2026-05-19  
**Status:** Approved — ready for implementation  
**Scope:** Sign metadata regeneration, shape/name enrichment, note generation, audit

---

## 1. Problem Statement

`src/data/signs.ts` is auto-generated by `scripts/generateSignsData.ts`, which infers sign shapes using numeric-range assumptions:

```
200-299 → "עיגול"   ← WRONG (instruction signs are rectangles)
400-499 → "עיגול"   ← WRONG for ~12 zone-marker signs
800-899 → "עיגול"   ← WRONG (these are road surface markings)
```

This causes the Order Form's auto-note field to display incorrect shape descriptions (e.g., "עיגול" for road markings, "עיגול" for rectangular zone signs). The bug propagates to the Order PDF and any future billing/audit logic that reads `row.notes`.

The correct shapes for every sign are documented in the official knowledge base at:  
`מקורות מידע/Sign/knowledge-base/categories/` (9 category files derived from the official Ministry of Transportation / Netivei Israel `לוח תמרורים.pdf`).

---

## 2. Source of Truth

**Primary source:** `מקורות מידע/Sign/knowledge-base/categories/*.md`  
These 9 files contain per-sign entries structured as:

```markdown
### Sign NNN — English Name

| Field | Value |
|---|---|
| **Number** | NNN |
| **Hebrew Term** | ... |
| **Shape / Color** | ... |
```

The `Shape / Color` field contains an English description that reliably encodes the canonical physical shape. This is the only authoritative basis for shape assignment.

**Image source:** `public/signs/` — the serving directory already populated with extracted images.

**No range-based fallback.** If a sign's shape cannot be derived from the knowledge base entry with confidence, set `needsReview: true` and omit a shape guess. A wrong shape is worse than a flagged unknown.

## 3. Data Model

### 3.1 `SignRecord` (updated interface in `src/types/order.ts` and `src/data/signs.ts`)

```typescript
export interface SignRecord {
  number: string;       // normalized key ("302", "129_b", "p401")
  imageFile: string;    // filename in public/signs/
  shape: string;        // canonical Hebrew shape token (see §4)
  name: string;         // official Hebrew sign name
  series: string;       // series label ("1xx", "2xx", … "9xx", "provisional")
  available: boolean;   // image file exists in public/signs/
  needsReview: boolean; // true if shape or name is uncertain
}
```

`series` and `available` are retained for backward compatibility. `needsReview` replaces the previous implicit uncertainty (missing shape = unknown).

### 3.2 `SignRow` (in `src/types/order.ts`) — no change

The `SignRow.notes` field continues to store free-text. The auto-fill behavior changes (§7) but the type is unchanged.

## 4. Shape Token Vocabulary

All shape values stored in `SignRecord.shape` must come from this list. No other values are permitted except `"לא ידוע"` paired with `needsReview: true`.

| Token | Hebrew | When to use |
|---|---|---|
| `"משולש"` | Triangle | 1xx warning signs; 9xx orange triangle variants |
| `"עיגול"` | Circle | 4xx prohibition circles; 9xx orange circle variants |
| `"מלבן"` | Rectangle | 2xx instruction signs; 4xx zone markers; 5xx; 6xx; 7xx lane panels; 9xx rectangle variants |
| `"משולש הפוך"` | Inverted triangle | Signs 303, 305 (give way) |
| `"מתומן"` | Octagon | Signs 302, 304 (STOP); sign 909 (orange STOP) |
| `"יהלום"` | Diamond | Sign 301 (priority road); sign 310 (priority road end) |
| `"סימון כביש"` | Road marking | All 8xx series; 5xx items 503, 504 (road surface markings) |
| `"מיוחד"` | Special | Barriers (403, 404); traffic signal housings (701–722); delineators/posts (928–935) |
| `"לא ידוע"` | Unknown | Only when knowledge base entry is absent or ambiguous — must pair with `needsReview: true` |

Provisional (p-prefix) signs inherit the shape of their base sign unless the category file explicitly documents a different shape.

## 5. Generator Script Design

### 5.1 New script: `scripts/generateSignsFromKnowledgeBase.ts`

Replaces `scripts/generateSignsData.ts`. Runs via `npm run generate:signs`.

**Pipeline:**

```
Step 1 — Parse knowledge base
  Read each of 9 category .md files from:
  מקורות מידע/Sign/knowledge-base/categories/

  For each sign entry (identified by "### Sign NNN"):
    - Extract number (from | **Number** | row)
    - Extract Hebrew term (from | **Hebrew Term** | row)
    - Extract shape description (from | **Shape / Color** | row)

Step 2 — Resolve shape token
  Apply keyword matching to the English shape description (§5.2)
  If no confident match → shape = "לא ידוע", needsReview = true

Step 3 — Clean Hebrew name
  Detect reversed RTL extraction artifacts (§5.3)
  If term appears reversed → flag needsReview = true, store as-is for manual fix

Step 4 — Build metadata map
  signKey → { name, shape, needsReview }
  For p-prefix entries: inherit shape from base sign if not explicitly stated

Step 5 — Scan public/signs/ filesystem
  List all sign_*.png files
  For each file, extract key via regex: sign_(p?\d+(_[bcd])?)\.png

Step 6 — Merge and emit
  For each image key found:
    Look up metadata map → get name, shape, needsReview
    If not in metadata → shape = "לא ידוע", needsReview = true, name = ""
  
  Emit src/data/signs.ts with enriched SignRecord entries
  Print audit summary to console (§6)
```

### 5.2 Shape Resolution — Keyword Matching

Applied to the English `Shape / Color` description string (case-insensitive). Rules are evaluated **in priority order** — first match wins:

| Priority | Keyword(s) in description | Assigned shape token |
|---|---|---|
| 1 | `inverted triangle` | `"משולש הפוך"` |
| 2 | `octagon` | `"מתומן"` |
| 3 | `diamond` | `"יהלום"` |
| 4 | `barrier`, `gate`, `chevron`, `delineator`, `post`, `signal` | `"מיוחד"` |
| 5 | `surface`, `marking`, `line`, `painted`, `road surface` | `"סימון כביש"` |
| 6 | `triangle` | `"משולש"` |
| 7 | `circle`, `circular` | `"עיגול"` |
| 8 | `rectangle`, `rectangular`, `panel`, `blue rect`, `green rect` | `"מלבן"` |
| 9 | *(no match)* | `"לא ידוע"` + `needsReview: true` |

This is purely data-driven from the knowledge base text. **No numeric ranges are used.**

### 5.3 Hebrew Name Extraction

The RTL PDF extraction produced some reversed word-order artifacts. Detection heuristic: if each Hebrew word in the term, when its characters are reversed, produces a valid Hebrew word pattern, the term is likely reversed.

Practical approach: extract the term as-is, then flag `needsReview: true` for terms that contain reversed-character word patterns. The audit report will list them. Corrections are made manually in a follow-up pass directly in `src/data/signs.ts` or via an override map.

The generator does **not** silently reverse or guess Hebrew terms — that would introduce new errors.

### 5.4 Series Labels

Derived from numeric prefix of the sign key (unchanged from old script):
- 1xx → `"1xx"`, 2xx → `"2xx"`, … 9xx → `"9xx"`, p-prefix → `"provisional"`

## 6. Audit Script

**New script:** `scripts/auditSignsData.ts`  
**Run via:** `npm run audit:signs`

Reports the following categories to stdout (and optionally to `scripts/audit-signs-report.md`):

| Audit category | Detection logic |
|---|---|
| Road markings with wrong shape | `series === "8xx"` AND `shape !== "סימון כביש"` |
| Suspicious 4xx circle assignments | `series === "4xx"` AND `shape === "עיגול"` AND `needsReview` |
| Missing Hebrew name | `name === ""` |
| Unknown shape | `shape === "לא ידוע"` |
| Reversed/malformed Hebrew term | Flagged via `needsReview` during generation |
| Image without metadata entry | Image files in `public/signs/` with no `SIGNS_DATA` entry |
| Metadata entry without image | `available === false` |
| Entries generated with fallback | `needsReview === true` for any reason |

The audit does not modify data. It only reports.

Also updates `signLookup.ts`'s existing `auditSignShapes()` function to include the new shape tokens in `KNOWN_SHAPES` so it stays aligned.

## 7. Auto-Note Generation (Order Form)

**File:** `src/components/OrderForm/SignRow.tsx`  
**Change:** line 39, inside the `useEffect` that fires when a sign lookup succeeds.

**Before:**
```ts
onChange({ imageUrl: `/signs/${record.imageFile}`, notes: record.shape, lookupStatus: "found" });
```

**After:**
```ts
const autoNote = record.name
  ? `${record.shape} — ${record.name}`
  : record.shape;
onChange({ imageUrl: `/signs/${record.imageFile}`, notes: autoNote, lookupStatus: "found" });
```

**Result examples:**
- Sign 402 → `עיגול — כניסה אסורה לכל כלי רכב`
- Sign 401 → `מלבן — כביש סגור`
- Sign 302 → `מתומן — עצור`
- Sign 801 → `סימון כביש — קו לבן רצוף`
- Sign 301 → `יהלום — כביש ראשי`
- Sign with `needsReview` and empty name → shape only (same as before, no regression)

The `notes` field remains fully editable by the user after auto-fill.

## 11. Known Limitations After This Fix

1. **Hebrew name reversal:** Some Hebrew terms extracted from the RTL PDF have word-order artifacts. These will be flagged `needsReview: true` in the generated file and listed in the audit report. Fixing individual names is a follow-up manual task.

2. **9xx work zone signs:** These use orange versions of standard shapes (orange triangle, orange circle, orange rectangle, etc.) per the knowledge base. The knowledge base entries describe them individually. The parser will derive correct shapes per-entry from the English descriptions. Sign 909 (orange STOP octagon) will correctly get `"מתומן"`.

3. **Symbol entries (ס-prefix):** The existing `src/data/signs.ts` does not contain symbol entries. Symbols are not traffic signs and are not used in the Order Form. They are out of scope for this fix.

4. **Sign 430 gap:** Per the completeness report, sign 430 has no image in the official source (gap confirmed from the PDF). It exists in the current `signs.ts` only if an image was previously sourced elsewhere. The generator will mark it `available: false` if no image is found in `public/signs/`.
